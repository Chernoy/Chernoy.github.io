3.final在java中的作用

  • 修饰一个引用

    • 引用为基本数据类型，则为常量不能修改
    • 为引用数据类型，如数组和对象，则数组和对象本身可以变，但是指向该数组或对象的地址不能改变
    • 为类的成员变量，则必须赋值否则编译报错

  • 修饰方法

    该方法称为最终方法，无法被子类重写，但是可以被继承。

  • 修饰类

    最终类，无法被继承

解释：继承可以理解为扩展，final修饰的方法，不能被子类重写覆盖，意味着无法改变该方法的实现，即无法自定义进行修改，然而无论有没有final修饰，都不影响类的继承性，子类会继承父类所有的非私有方法，包括被final修饰的方法，但是如果子类中有方法与父类中被final修饰的方法重名，则会编译报错。另一方面，对于类而言，final 修饰的类是无法被继承的。这意味着其他类无法扩展（继承）一个被 final 修饰的类。这通常用于表示某个类的设计是最终的，不允许其他类继承该类，以确保该类的行为和实现不会被改变。

30、什么是 Java 的内存模型?

在了解什么是 Java 内存模型之前，先了解一下为什么要提出 Java 内存模型。

之前提到过并发编程有三大问题

CPU 缓存，在多核 CPU 的情况下，带来了可见性问题
操作系统对当前执行线程的切换，带来了原子性问题
译器指令重排优化，带来了有序性问题
为了解决并发编程的三大问题，提出了 JSR-133，新的 Java 内存模型，JDK 5 开始使用。

简单总结下

Java 内存模型是 JVM 的一种规范
定义了共享内存在多线程程序中读写操作行为的规范
屏蔽了各种硬件和操作系统的访问差异，保证了 Java 程序在各种平台下对内存的访问效果一致
解决并发问题采用的方式：限制处理器优化和使用内存屏障
增强了三个同步原语（synchronized、volatile、final）的内存语义
定义了 happens-before 规则

49、怎么确保一个集合不能被修改？

52、ConcurrentHashMap(JDK1.8)为什么要使用synchronized而不是如ReentranLock这样的可重入锁？

我想从下面几个角度讨论这个问题：

（1）锁的粒度

首先锁的粒度并没有变粗，甚至变得更细了。每当扩容一次，ConcurrentHashMap的并发度就扩大一倍。

（2）Hash冲突

JDK1.7中，ConcurrentHashMap从过二次hash的方式（Segment -> HashEntry）能够快速的找到查找的元素。在1.8中通过链表加红黑树的形式弥补了put、get时的性能差距。
JDK1.8中，在ConcurrentHashmap进行扩容时，其他线程可以通过检测数组中的节点决定是否对这条链表（红黑树）进行扩容，减小了扩容的粒度，提高了扩容的效率。

下面是我对面试中的那个问题的一下看法。

为什么是synchronized，而不是ReentranLock

（1）减少内存开销

假设使用可重入锁来获得同步支持，那么每个节点都需要通过继承AQS来获得同步支持。但并不是每个节点都需要获得同步支持的，只有链表的头节点（红黑树的根节点）需要同步，这无疑带来了巨大内存浪费。

（2）获得JVM的支持

可重入锁毕竟是API这个级别的，后续的性能优化空间很小。
synchronized则是JVM直接支持的，JVM能够在运行时作出相应的优化措施：锁粗化、锁消除、锁自旋等等。这就使得synchronized能够随着JDK版本的升级而不改动代码的前提下获得性能上的提升。

53concurrentHashMap和HashTable有什么区别

ConcurrentHashMap 和 HashTable 都是用于实现线程安全的哈希表（Hashtable）的类，但它们在设计和性能上存在一些区别。

  1. 线程安全性：
    • HashTable：是早期 Java 集合框架中提供的线程安全的哈希表实现。它的所有方法都是同步的，也就是说，在多线程环境下，多个线程可以安全地访问和修改 HashTable 的实例。
    • ConcurrentHashMap：是 JDK 1.5 引入的线程安全的哈希表实现，相较于 HashTable，ConcurrentHashMap 采用了更精细的锁机制和分段锁的设计。它在多线程环境下提供更好的并发性能，允许多个线程同时进行读取操作，并且仅在写入操作时进行必要的同步。
  2. 锁机制：
    • HashTable：使用一把全局锁（synchronized）来锁定整个 HashTable 对象，在多线程环境下可能会出现较高的竞争和性能瓶颈。
    • ConcurrentHashMap：采用了分段锁（Segment）的设计，内部将哈希表分成多个段（Segment），每个段维护一部分数据，不同段之间的数据可以并行访问，读写操作只锁定对应的段，多线程读取操作时不会互斥，提供了更好的并发度和性能。
  3. 迭代器支持：
    • HashTable：使用迭代器（Iterator）遍历 HashTable 的时候，需要获取全局锁，阻塞其他线程的写操作。
    • ConcurrentHashMap：支持并发的键-值遍历，通过使用 ConcurrentHashMap 的特殊迭代器，允许多线程并发地读取，而不会阻塞写入操作。
  4. 允许空键值：
    • HashTable：不允许键或值为 null。
    • ConcurrentHashMap：允许键和值为 null。

总体而言，ConcurrentHashMap 在多线程环境下的性能更好，因为它利用了更细粒度的锁机制和并发技术，减少了竞争和互斥。如果使用 Java 5 或更高版本，推荐使用 ConcurrentHashMap 来替代 HashTable。然而，如果需要保证传统的线程安全性，且不关注性能，仍然可以使用 HashTable。

55.请谈谈 ReadWriteLock 和 StampedLock

ReadWriteLock（读写锁）和 StampedLock（标记锁）都是用于并发控制的锁机制，但在实现和使用方式上有一些区别。

  1. ReadWriteLock（读写锁）：
    • ReadWriteLock 允许多个线程同时读取共享资源，但只允许一个线程写入共享资源。
    • ReadWriteLock 接口提供了两个关键的方法：readLock() 和 writeLock()，用于获取读锁和写锁。
    • 读锁（共享锁）可以被多个线程同时获取，只要没有线程持有写锁。读锁之间是不互斥的，允许并发读取共享资源，因此适用于读多写少的场景。
    • 写锁（排它锁）只能被一个线程获取，当线程持有写锁时，其他线程无法获取读锁或写锁。写锁用于修改共享资源，因此对共享资源的修改是互斥的，适用于写操作较频繁的场景。
  2. StampedLock（标记锁）：
    • StampedLock 也支持读写分离的并发访问，类似于 ReadWriteLock，但提供了更高级别的功能。
    • 在 StampedLock 中，锁的状态由一个 long 类型的标记表示，称为 stamp。
    • StampedLock 提供了三种具体的锁：读锁、写锁和乐观读锁。
    • 读锁（共享锁）和写锁（排它锁）的使用方式与 ReadWriteLock 类似，但乐观读锁是一种无锁操作，不阻塞其他线程的锁获取。
    • 乐观读锁的获取方式是用 tryOptimisticRead() 方法，获取到一个标记，并在后续读操作中验证标记的有效性，如果标记无效，则需要重新获取锁。
    • StampedLock 还提供了手动的锁降级和锁升级操作，即从写锁降级为读锁或从读锁升级为写锁。

总体而言，ReadWriteLock 适用于读多写少的场景，提供了明确的读锁和写锁，适合对共享资源进行高并发读取和独占性写操作的控制。而 StampedLock 则提供了更灵活的锁方式，包括读锁、写锁和乐观读锁，适用于读多写少且具备无锁操作和锁降级/升级需求的场景。选择使用哪种锁机制应根据具体的应用需求和并发访问模式来确定。

56、线程的run()和start()有什么区别？

每个线程都是通过某个特定Thread对象所对应的方法run()来完成其操作的，run()方法称为线程体。通过调用Thread类的start()方法来启动一个线程。

start() 方法用于启动线程，run() 方法用于执行线程的运行时代码。run() 可以重复调用，而 start() 只能调用一次。

start()方法来启动一个线程，真正实现了多线程运行。调用start()方法无需等待run方法体代码执行完毕，可以直接继续执行其他的代码； 此时线程是处于就绪状态，并没有运行。 然后通过此Thread类调用方法run()来完成其运行状态， run()方法运行结束， 此线程终止。然后CPU再调度其它线程。

run()方法是在本线程里的，只是线程里的一个函数，而不是多线程的。 如果直接调用run()，其实就相当于是调用了一个普通函数而已，直接待用run()方法必须等待run()方法执行完毕才能执行下面的代码，所以执行路径还是只有一条，根本就没有线程的特征，所以在多线程执行时要使用start()方法而不是run()方法。

58.synchronized的工作原理

synchronized 是 Java 中用于实现线程同步的关键字。它可以用于修饰代码块或方法，或者使用在 static 方法或静态代码块中。当多个线程试图同时访问被 synchronized 修饰的代码块或方法时，只有一个线程能够获得锁，其他线程将进入阻塞状态，直到获取到锁的线程释放锁。

在 Java 中，每个对象都有一把内部锁（即监视器锁或互斥锁），这把锁与对象实例关联。只有持有锁的线程才能够执行被 synchronized 修饰的代码块或方法。当一个线程进入一个 synchronized 代码块时，它会尝试获取对象实例的锁。如果锁没有被其他线程占用，该线程将获得锁，并执行代码块或方法。在执行结束时，它会释放锁，使其他线程可以竞争获取同样的锁。

synchronized 关键字实现了以下特性：

  1. 互斥性：同一时间只允许一个线程进入被 synchronized 修饰的代码块或方法，保证了线程间的互斥执行。
  2. 可见性：线程在释放锁之前，对共享变量所做的修改对其他线程可见。这是因为进入 synchronized 代码块时会将变量从主存中读取到线程的工作内存中，执行结束时会将修改的结果刷新回主存。
  3. 顺序性：线程在获取锁时，会按照获取锁的顺序进行排队，保证了代码块或方法的执行顺序。
  4. 非公平性：JVM 对 synchronized 关键字的实现通常是非公平的，即不保证等待的线程按照先后顺序获得锁。

需要注意的是，synchronized 关键字仅作用于同一个锁对象上的线程，对不同锁对象的线程没有影响。此外，synchronized 关键字在使用时需要谨慎，过度使用或不当使用可能导致死锁、性能问题或并发性下降。在 Java 5 之后，JDK 提供了更多并发工具，如 ReentrantLock 和 Lock 接口，可以提供更灵活和细粒度的锁控制。

60.为什么 wait(), notify()和 notifyAll()必须在同步方法或者同步块中被调用

wait(), notify(), 和 notifyAll() 方法必须在同步方法或同步块中被调用是因为它们是用于线程之间的协作和通信，而线程之间的协作需要通过共享的监视器锁来实现。

当一个线程执行一个对象的 wait() 方法时，它会释放该对象上的锁，并且进入等待状态，直到其他线程调用相同对象的 notify() 或 notifyAll() 方法来通知等待的线程。类似地，notify() 方法用于唤醒任意一个等待线程，而 notifyAll() 方法用于唤醒所有等待线程。

为了确保线程之间正确的协作，wait(), notify(), 和 notifyAll() 方法必须在已经获取了对象的锁的线程上进行调用。这是因为只有持有锁的线程才能够进入被 synchronized 修饰的代码块或方法，这些方法需要在获取到对象的锁之后才能执行。

如果在没有持有对象锁的线程上调用这些方法，即在非同步方法或非同步块中调用，会导致 IllegalMonitorStateException 异常。

因此，为了保证线程之间的有效协作和正确的同步，wait(), notify(), 和 notifyAll() 方法需要在同步方法或同步块中被调用，这样能够确保线程在正确的上下文中对共享资源进行协调和通信。

61、Java 如何实现多线程之间的通讯和协作？

  1. wait()、notify() 和 notifyAll()：通过调用对象的 wait() 方法，一个线程可以进入等待状态，直到其他线程调用相同对象的 notify() 或 notifyAll() 方法来唤醒等待的线程。这种方式是基于对象的锁机制实现的，通常需要与 synchronized 关键字一起使用。
  2. Condition 接口：java.util.concurrent.locks.Condition 接口提供了更灵活的线程通讯机制，不同于对象的 wait() 和 notify()，Condition 提供了 await()、signal() 和 signalAll() 方法，可以在使用 Lock 对象（如 ReentrantLock）时进行调用。
